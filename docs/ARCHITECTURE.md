# Архитектура

## Слои
- **UI (`ui/`)** — экраны Kivy (`.py` + `.kv`), ScreenManager.
- **Сервисы (`services/`)** — бизнес-логика: посещаемость, отчёты, диаграммы,
  генерация `.ics`, напоминания.
- **Данные (`db.py`, `models.py`)** — доступ к SQLite, нормализация, CRUD.

## Поток данных
UI вызывает функции сервисов и `db`. `db` — единственная точка доступа к SQLite
(соединение через контекстный менеджер, `PRAGMA foreign_keys=ON`). Отчёты
считаются на лету из таблиц `lessons`/`attendance`.

## Экраны
`home` → `groups` / `students_bulk` / `subjects` / `mark` / `reports`.

## Принятые инженерные решения (и обоснования)
1. **Kivy без KivyMD.** Меньше зависимостей → надёжнее облачная сборка p4a,
   меньше размер APK. Достаточно стандартных виджетов.
2. **Диаграммы на Kivy Canvas, а не matplotlib.** matplotlib/garden под
   python-for-android часто ломает сборку и сильно раздувает APK. Простые
   столбчатые/полосовые диаграммы рисуются на Canvas без зависимостей.
   (F6 допускает такой выбор явно.)
3. **Единица учёта — «занятие».** Везде считаем проведённые занятия, а не часы.
   `subjects.planned_lessons` — план в занятиях; факт = `COUNT(conducted=1)`.
4. **Неповторяемость через БД.** UNIQUE-индексы + нормализация имён
   (`name_norm`) защищают от дублей даже при гонках; UI лишь предлагает выбор
   из списка.
5. **Напоминания — при открытии приложения (основной механизм).** Фоновые
   точные будильники Android ненадёжны/ограничены ОС, поэтому критичная логика
   завязана на запуск приложения; `plyer`-уведомления вторичны.
6. **Календарь — только `.ics`.** Никаких API: генерируем RFC 5545 файл,
   пользователь импортирует вручную. Ноль сетевых вызовов, ноль ключей.

## Схема БД
Кратко: `groups 1—* students`, `groups 1—* subjects`,
`subjects 1—* schedule_slots`, `subjects 1—* lessons`,
`lessons 1—* attendance *—1 students`. Подробно — [DB_SCHEMA.md](DB_SCHEMA.md).
